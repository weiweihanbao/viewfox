⚙️ Vibe Coding 开发实践

一、Vibe Coding 是什么？

定义

Vibe Coding = OpenSpec + Superpowers 驱动的AI协同开发范式

将传统"全人工编码"升级为"人机协同"模式：

  • 人：负责需求理解、架构设计、核心算法、边界判断、最终把控
  • AI（Claude）：负责代码定位、调用链梳理、样板代码生成、重复性改造、文档生成

对比传统开发

传统开发：需求 → 设计 → 编码 → 测试（全人工）

Vibe Coding：需求 → 设计 → [AI 辅助编码] → 人工校验 → 测试

三、实战案例：字节码解析模块

人工部分（30%）

工作内容：

  1. 确定技术选型：选择java-all-call-graph
  2. 设计H2数据模型（表结构）
  3. 核心算法设计：DFS递归逻辑
  4. 黑名单判定规则

AI辅助部分（70%）

Prompt示例：

请帮我生成以下代码：

1. H2数据库访问层（Repository）
   - 表：jacg_method_call
   - 查询方法：
     • queryCallees(String methodHash): 查询下游方法
     • queryCallers(String methodHash): 查询上游方法
   - 使用MyBatis，返回List<MethodCallDO>

2. 方法信息查询接口
   - 表：jacg_method_info
   - 根据methodHash查询方法签名、类名、行号

请生成完整的Mapper接口、XML文件、DO实体类。

AI输出：完整的Mapper接口 + XML映射 + DO实体类（约500行代码）

人工工作：校验SQL正确性、补充异常处理、添加日志

五、Vibe Coding 最佳实践

✅ 适合交给AI的任务

  • 样板代码生成：Controller、Service、DAO、DTO、配置类
  • 调用链定位：基于关键词快速找到相关代码位置
  • 重复性改造：批量修改方法签名、统一异常处理
  • 文档生成：API文档、使用手册、测试用例文档
  • 代码解释：解释复杂的第三方库用法、框架机制

❌ 必须人工负责的任务

  • 业务规则理解：如"什么是零流量接口"、"哪些方法必须保护"
  • 核心算法设计：级联删除的递归逻辑、可删除性判定
  • 架构决策：技术选型、模块划分、数据模型设计
  • 边界与异常处理："编译失败如何回滚"、"并发任务如何隔离"
  • 代码审查与测试：AI生成的代码必须人工校验

六、Prompt 编写技巧（面试重点）

❌ 错误示例（模糊、无上下文）

帮我写一个方法查询数据库

✅ 正确示例（明确、有上下文）

请帮我生成MyBatis Mapper接口和XML文件：

【表结构】
表名：jacg_method_call
字段：caller_hash (VARCHAR), callee_hash (VARCHAR),
      callee_method (VARCHAR)

【查询需求】
根据caller_hash查询该方法调用的所有下游方法（callee）
返回类型：List<MethodCallDO>

【要求】
1. Mapper接口名：MethodCallMapper
2. 方法名：queryCallees
3. 使用@Param注解
4. XML文件格式规范（4空格缩进）

八、面试回答要点

面试官可能这样问：

Q: "你们用了AI协同开发，具体怎么做的？效果如何？"

回答框架：

  1. 背景："采用Vibe Coding范式，人负责核心算法和架构，AI辅助样板代码生成"
  2. 分工："核心DFS算法100%人工，样板代码（CRUD、Mapper、DO）90%由AI生成"
  3. 流程："我给AI写详细的Prompt（包含表结构、查询需求、代码规范），AI生成代码后我人工校验"
  4. 效果："25天工作量压缩到10天，开发效率提升60%，代码注释覆盖率从60%提升到85%"
  5. 关键："AI是助手不是决策者，核心代码必须人工Review，上线前完整测试验证"
