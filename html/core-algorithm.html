💡 核心算法详解

一、级联删除算法（DFS深度优先）

算法目标

从零流量接口出发，递归分析所有只被该删除链路调用的方法，构建完整的可删除方法集合。

核心思想： 如果方法M的所有调用方都在"待删除集合"中， 且M不在黑名单中，那么M也可以删除。 递归应用这个规则，直到无新方法可加入。

算法伪代码

// 输入：零流量接口方法哈希
// 输出：可删除方法集合

Set<String> deletableSet = new HashSet<>();
Set<String> visited = new HashSet<>();

function dfs(methodHash):
    if visited.contains(methodHash):
        return
    visited.add(methodHash)

    // 将当前方法加入可删除集合
    deletableSet.add(methodHash)

    // 查询该方法调用的所有下游方法
    callees = queryCallees(methodHash)

    for each callee in callees:
        // 查询callee的所有调用方
        callers = queryCallers(callee)

        // 判断callee是否只被删除链路调用
        onlyCalledByDeletePath = true
        for each caller in callers:
            if not deletableSet.contains(caller):
                onlyCalledByDeletePath = false
                break

        // 如果callee只被删除链路调用，且不在黑名单
        if onlyCalledByDeletePath and not isInBlacklist(callee):
            dfs(callee)  // 递归分析

二、复杂场景处理

场景1：线性调用链

graph LR A[方法A
零流量接口] --> B[方法B] B --> C[方法C] C --> D[方法D] style A fill:#ffcdd2 style B fill:#ffcdd2 style C fill:#ffcdd2 style D fill:#ffcdd2

分析过程：

  1. 删除A，A加入deletableSet
  2. B只被A调用 → B可删，加入deletableSet
  3. C只被B调用 → C可删，加入deletableSet
  4. D只被C调用 → D可删，加入deletableSet

结果：A、B、C、D 全部删除

场景2：共享依赖

graph LR A[方法A
零流量接口] --> C[方法C
共享方法] B[方法B
正常使用] --> C style A fill:#ffcdd2 style B fill:#c8e6c9 style C fill:#c8e6c9

分析过程：

  1. 删除A，A加入deletableSet
  2. C被A和B调用，B不在deletableSet
  3. C不满足"只被删除链路调用" → C保留

结果：只删除A，C保留

场景3：循环依赖

graph LR A[方法A
零流量接口] --> B[方法B] B --> C[方法C] C --> B style A fill:#ffcdd2 style B fill:#ffcdd2 style C fill:#ffcdd2

问题：普通DFS会陷入死循环

解决方案：visited集合记录已访问节点

  1. 访问A → A加入visited
  2. 访问B → B加入visited
  3. 访问C → C加入visited
  4. 再次访问B → 检测到B在visited中，直接return

结果：算法正常结束，A、B、C都可删

四、黑名单判定逻辑

判定流程

判定顺序（优先级从高到低）： 1. 白名单检查 if isInWhitelist(method): return DELETABLE // 强制删除 2. 黑名单检查 if isInBlacklist(method): return PROTECTED // 必须保留 3. 调用链分析 if onlyCalledByDeletePath(method): return DELETABLE else: return KEEP

六、面试回答要点

高频问题1：为什么用DFS而不是BFS？

回答：

  1. 调用链特点：方法调用链通常较深（10-20层），DFS更符合调用栈的自然结构
  2. 空间复杂度：DFS只需维护当前路径的栈空间，BFS需要队列存储所有同层节点
  3. 提前剪枝：DFS遇到黑名单方法可立即停止该分支，BFS需要等待整层结束
  4. 代码简洁：DFS递归实现更直观，易于理解和维护

高频问题2：如何保证不误删？

回答（多重保障）：

  1. 调用链分析：只删除"仅被删除链路调用"的方法
  2. 黑名单机制：保护支付类、核心链路、框架方法（42个方法被保护）
  3. 编译校验：删除后立即Maven编译，失败自动回滚
  4. 测试流水线：自动触发156个单元测试+42个集成测试
  5. 人工审批：预览报告需Tech Leader审查

结果：上线后3个月，误删率0%，线上故障0起

高频问题3：循环依赖怎么处理？

回答：

  1. visited集合：记录已访问的方法，检测到重复访问立即return
  2. 不影响结果：循环依赖的方法如果都只被删除链路调用，仍然全部可删
  3. 实际案例：在A→B→C→B这种场景下，算法正确识别A、B、C都可删
